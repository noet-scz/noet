# 05 — Роадмап

Переход от «реестр хранит всё» к «конструктор из примитивов». Каждая фаза: цель, стек, todo,
выходной тест (как у концепции §11 — проверяемый, не «сделано на глаз»), анти-платформенная
проверка. Оценка в **единицах работы**, не в календаре.

Порядок не жёсткий, но P1→P2 это фундамент федерации; без них «забрать с собой» формально.

---

## P0 — Сделано

Личность (Nostr-ключ в расширении, NIP-07), имена в реестре, страницы и проекты в IPFS,
sandbox-рендер с JS, реле-чат, репутация-ёмкость + делегации доступа (gov.mjs), страницы
people/dev, право форка как `/api/export`. Владелец экземпляра назван.

---

## P1 — SDK примитивов (поверхность конструктора) — СДЕЛАНО

**Цель:** превратить window.noet (me/publish/query) в полный SDK пяти примитивов, чтобы
снаружи собирался любой продукт без изменения ядра.

**Стек:** `ext/noet-primitives.js` — единый источник правды, наращивает window.noet поверх
ядра; работает И в content-script (обычные страницы), И в sandbox-рендере (view.js передаёт
исходник в render.html). Nostr kinds: 0/1/3/7/30023/1063/30078/31000–31002. Публичные wss-реле.

**Todo:**
- [x] `noet.identity.me/get/follows` (vouch/grant остаются на people.nt — чувствительны, не из произвольного контента)
- [x] `noet.space.create/get/list` (kind 31000, `type`-поле строкой)
- [x] `noet.artifact.publish/query/get` (мелкое→событие; `cid`-поле для крупного→IPFS)
- [x] `noet.relation.add/query` (reply NIP-10 / react / mention / rel; e/p/a/t-теги)
- [x] `noet.policy.set/get/can` (kind 31001; `can` — клиентская подсказка)
- [x] версия SDK (`noet.version`) + полная дока на dev.nt (живые сниппеты)
- [x] шаблон-демонстратор `sites/guestbook/` (Space feed + Artifact + Relation reply)

**Выходной тест:** ✓ гостевая книга собрана целиком на примитивах (~80 строк, без сервера,
без правок ядра); 15-проверочный CDP-харнес round-trip через мост sandbox-рендера зелёный.
Открывается у другого через те же публичные реле (см. [03-federation.md](03-federation.md)).

**Анти-платформа:** ✓ даём кубики, не продукт. Ни один вызов ядра не знает слова «гостевая
книга»; vouch/grant сознательно НЕ отданы произвольному контенту (остаются явным действием
на people.nt).

## P2 — Данные из реестра в события (федерация по-настоящему) — В РАБОТЕ

**Цель:** личность, контент, политика перестают жить в реестре-силосе; экземпляр
становится ВИДОМ. Это включает «забрать с собой без пересоздания» ([03-federation.md](03-federation.md)).

**Стек:** публичные реле как дом данных; IPFS + пиннинг «своих»; реестр = `name→координата`
+ сидер. Адресуемые события NIP-33 для страниц/Space/Policy.

**Todo:**
- [x] Страница `.me` = подписанное событие 31002 (d=имя, content→CID) дозаписывается на публичные реле при публикации (account.js); реестр остаётся индексом+сидером
- [x] Резолвер расширения читает указатель с реле (relayRecordCid): мгновенно рендерит CID из индекса, фоном берёт более свежий с реле (без регресса скорости)
- [x] Поручительства/доступы (8001/8002/8010/8011) дозаписываются на публичные реле (people.html) — личность и политика портативны
- [x] Список реле сообщества опубликован в config (часть «корня»)
- [x] `/api/export` остаётся как слепок
- [x] Профиль (kind 0) уходит на публичные реле при сохранении и засевается при регистрации (account.js) — личность переносима
- [x] people.nt читает имена/аватары участников с публичных реле (kind 0) — личность отображается из событий, не из реестра
- [x] Честный статус «где живут данные / кто хранит» на people.nt (durability §4.2 без вранья)
- [ ] Независимые пиннеры у участников (нужен узел на стороне участника) — реальная несокрушимость, отдельная фаза
- [ ] Резолв `name→владелец` тоже как событие-заявка (сейчас из индекса-зеркала) → ведёт в P5/OTS

**Выходной тест:** ✓ доказан примитив федерации: указатель 31002, опубликованный на ЖИВЫЕ
публичные реле (damus/nos.lol), читается НЕЗАВИСИМЫМ соединением с тем же CID — другой
экземпляр/резолвер видит данные БЕЗ сервера A (node-тест зелёный). Полный второй экземпляр —
следующая итерация.

**Анти-платформа:** ✓ убираем силос — антиплатформенно по определению.

## P3 — Шаблоны-продукты (демонстрация North Star) — СДЕЛАНО (ядро)

**Цель:** показать конструктор в деле: готовые открываемые шаблоны, каждый — композиция
примитивов, каждый форкается и кастомизируется. Это не фичи ядра, это контент.

**Стек:** статические сайты в IPFS, используют window.noet SDK. Деплой через существующий
publish-dir.

**Todo:**

- [x] Форум (`sites/forum/`) — темы (Artifact subject) + ответы (Relation reply), навигация по темам
- [x] Блог (`sites/blog/`) — статьи (Artifact, лонгрид 30023) + чтение
- [x] Галерея шаблонов на dev.nt с исходниками и инструкцией «форкни и задеплой»
- [x] Маркет (`sites/market/`) — лоты (Artifact JSON) + отзывы (Relation reply)
- [x] Вики (`sites/wiki/`) — страницы по ключу (Artifact 30078, версии) + [[ссылки]]-Relation

**Выходной тест:** ✓ 6 разных продуктов (книга/форум/блог/маркет/вики/дашборд+игра) на ОДНИХ
примитивах, ядро не выросло ни на один kind. UI форума и вики проверены CDP end-to-end
(вики: создание, состояние [[ссылок]], загрузка в редактор, версии).

**Анти-платформа:** ✓ шаблоны снаружи, ни одного нового kind ядра. Если шаблон потребовал
правки ядра — стоп, это была платформа.

**Стек:** расширение (онбординг, управление ключами), web UI.

## P5 — Имена без сервера (OpenTimestamps)

**Цель:** убрать последнюю серверную власть — судью очерёдности имён. Имя-заявка с
OTS-якорем; победитель — самая ранняя; считает клиент.

**Стек:** javascript-opentimestamps (браузер, бесплатные calendar-серверы), заявки как
события на реле, клиентский расчёт реестра.

**Todo:**
- [ ] Заявка имени = подписанное событие + OTS-пруф
- [ ] Клиентский резолвер: собрать заявки, проверить подписи+OTS, выбрать раннюю
- [ ] Оптимистичное использование имени до анкеринга; разрешение редких коллизий по якорю
- [ ] Снапшоты лога имён в IPFS для durability

**Выходной тест:** имя резолвится и без живого реестра-сервера; две конфликтные заявки
разрешаются одинаково на разных клиентах.

**Анти-платформа:** ✓ убирает спрятанного суверена корня имён в нейтральные часы.

## P6 — Приватность и масштаб

**Цель:** приватные пространства; устойчивость на росте.

**Стек:** NIP-44 (шифрование), NIP-87 (закрытые группы); снапшоты/лайт-клиенты; деградация
до меша как аварийный режим (§9.1).

**Todo:**

- [ ] Лайт-клиент: не тянуть весь лог, индексы/снапшоты

**Выходной тест:** закрытая группа читается только участниками; клиент работает, не
загружая весь мир.

**Анти-платформа:** ✓ приватность это Policy+шифр над теми же примитивами, не новый продукт.
